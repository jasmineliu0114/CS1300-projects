Accessible Components: Radio Buttons

This project explores the design of radio buttons and how changes in the design make trade-offs between learnability, memorability, efficiency, and accessibility for mouse, keyboard, touchscreen, and screen reader users.

Inputs

To get a sense of radio button characteristics, we explore and compare inputs from mouse, keyboard, and touchscreen users for three applications: Google Forms, Amazon Checkout, and Outlook Settings. This includes a focus on what users can and cannot do, functionalities that are learnable, memorable, and efficient, and comparisons between the three applications on interaction and accessibility.

Mouse/Trackpad
  • Hovering highlights the button.
  • Click+release the button or the label space selects and unselects the button. Clicking on the label instead of the button is not immediately obvious but is easily learnable. This feature is also efficient, as the user doesn't have to move their mouse all the way to the button to select.
  • Double-clicking the label text highlights a word and triple-clicking highlights the whole label, but these actions also activate/deactivate the button with each click.
  • Clicking and dragging also highlights label text and activates/deactivates the button. This could be less efficient since it can select/unselect the button when undesired.
  • Highlighted text can be copy and pasted to clipboard.
  • Click+release the button or the label space selects and unselects the button.
  • Unlike Google Forms, hovering over a button does not highlight the button or do anything at all.
  • Users cannot click to unselect a payment method. In the context of this application, this is efficient because one of the payment methods list must be selected in order to complete a purchase.
  • Double/triple-clicking and clicking+dragging the mouse highlights label text and selects the corresponding radio button. However, there is no deactivating of the button that occurs like in the case of Google Forms.
  • Highlighted text can be copy and pasted to clipboard.
  • Like the other two applications, click+release the button or the label space selects and unselects the button.
  • Hovering makes the label text slightly darker.
  • Unlike the other two applications, the user cannot select text in the label, neither by double/triple-clicking or by clicking and dragging the mouse.
  • Like the Amazon checkout payment application, users cannot click a radio button to deactivate the selection. Again, this is efficient because one of the options has to be selected.
  • Like the other two applications, the circular buttons make the action of selecting item easy to learn and memorable, as it is an interface that is consistent across all different applications.
Keyboard
  • If no items are selected, tab highlights the first radio button with a light purple circle. Otherwise, the selected radio button is highlighted.
  • To navigate to each radio button, the user uses the up/left and down/right arrow keys. This action also activates the radio button that the user moves to. This is not immediately obvious and required a few different keyboard button presses to figure out but is easy to learn and memorize once discovered. It is also pretty efficient, since it moves and selects in one action.
  • Clearing a selection requires pressing tab to highlight the Clear selection button and hitting enter. This is efficient in the sense that it requires fewer key presses than navigating over to the radio button a user wants to deactivate but less efficient in terms of having to leave the radio button space in order to select the Clear selection button.
  • Tab highlights the selected radio button with a circle outlining the button.
  • Like Google Forms, the up/left and down/right keys navigates to and selects the upper/lower radio button.
  • As previously mentioned, users cannot deactivate radio buttons, so there is no keyboard action for this either.
  • Tab highlights the selected radio button with a rectangle outlining both the radio button and its label.
  • Up/left and down/right arrows navigate and select the upper/lower radio buttons, like in the other two applications. Also like the other two, pressing up/left on the top item or down/right on the last item wraps the navigation to the other end of the list, which also make for more efficient navigation.
  • Again, users cannot deactivate radio buttons, so there is no keyboard action for this either.
iPhone Touchscreen
  • Tapping the button or the label space selects and unselects the button.
  • Long tapping the label text highlights a word. Unlike double-clicking the mouse, this action does not activate or deactivate the associated radio button.
  • The mobile version of the Amazon checkout payment page does not have radio buttons.
  • The mobile version Outlook does not have this option in Settings.

  • Unselected buttons are black-outlined circles. A selected button is filled with a slightly smaller colored button, and the original black circle outline also becomes colored. In terms of accessibility, this makes the selected button much more contrasted than the rest.
  • Hovering over the Clear selection button highlights the button in a light grey box. Clicking darkens the highlight box.
  • Focus order when pressing tab: Radio button → Clear selection button (if a radio button is selected) → Submit

A selected button is filled with a slightly smaller colored button, and the original black circle outline also becomes colored. However, unlike the other two applications, the selected radio button and corresponding label are also highlighted by a translucent box. Considering that a correct selection of a payment method is significantly more important than in the other two cases, this adds to the accessibility of the Amazon checkout page by adding addition contrast to selected items.
  • Hovering over both a button or its label makes the label text slightly darker. This is not a very obvious difference, but gives the user a slight indication that what they are hovering over may be interactive.

Keyboard
  • The status of pressing tab for radio buttons is indicated by the same circle highlight as hovering.
  • When the tab key changes status to the Clear selection button, the button is also highlighted by a light grey box, like with hovering.
  • When pressing up/left or down/right arrow keys to move to and select radio buttons, the selected button is both highlighted by a circle to indicate status and is filled in with the smaller colored circle mentioned in the mouse section.
  • Tab highlights the selected radio button with a circle outlining the button.
  • Similar to the other two applications, navigation and button selection using the up/left and down/right arrow keys is also indicated with highlighting the current radio button with a circle outlining the button and a smaller circle filling in the selected radio button. Like mentioned in the mouse section, this also has the addition colored box around both the radio button and label text.
  • Tab highlights the selected radio button with a rectangle outlining both the radio button and its label.
  • Again, navigation and button selection using the up/left and down/right arrow keys is indicated with highlighting the current radio button and label with a rectangle outline and a smaller circle filling in the selected radio button.
iPhone Touchscreen
  • Touchscreen has the same indication of a selected radio button as on a laptop.
  • Tapping on the button itself highlights the button in a larger transparently colored circle to indicate cursor status, but tapping on the label only selects/unselects and does not highlight the radio button.
  • The mobile version of the Amazon checkout payment page does not have radio buttons.
  • The mobile version Outlook does not have this option in Settings.
MacOS VoiceOver
  • Navigating the focus to an item announces the component type (radio button), the radio button status (selected or not), and the label text.
  • If a button is unselected, VoiceOver also tells the user which keyboard commands to press in order to select/unselect it.
  • In addition to announcing the radio button label and status, VoiceOver also tells the user which order radio button it is on (ex: radio button 1 of 3). This increases the accessibility from Google Forms since it allows the user to keep track of which number radio button they are on.
  • VoiceOver for the Outlook radio buttons also announces the button label, status, and order out of all the options.
  • Since there is no option of deselecting a radio button, VoiceOver doesn’t give it as an option like for Google Forms.

Revised Google Forms Radio Button Component: Mouse User

Learnability:

  • Hover state improves discoverability
  • Click and hold is not as intuitive

Memorability

  • Unique interaction makes it stand out
  • Uncommon behavior might be forgotten

Efficiency

  • Prevents accidental selections (unintentional clicks or double-clicking to highlight text)
  • Slower interaction and more effort needed

Accessibility

  • Hover state improves visual clarity
  • Holding may be difficult for users with limited dexterity

Revised Google Forms Radio Button Component: Keyboard User

Learnability:

  • More structured interaction
  • Clearer feedback for first-time users

Memorability

  • More distinct interaction flow
  • Less conventional behavior

Efficiency

  • Eliminates the need to exit the component to clear selection
  • Extra key presses

Accessibility

  • More deliberate selection
  • Focus state improves visual clarity

Reflection

After watching Kat Holmes' Inclusive Design, we reflect on the observations and revision made above. Here are the key points regarding usability and accessibility of the different design choices.

  • The radio button components provided clear visual feedback using hover effects, focus indicators, and active states, which help to improve discoverability.
  • Keyboard navigation was also clear, as tab and arrow key interactions allowed keyboard users to interact effectively.
  • The redesigned radio buttons are designed with even stronger visual indicator, not only highlighting elements more clearly for users who may have low vision, but also making it clearer that both radio buttons and their associated labels can be clicked to select/deselect.
  • Standardizing the user flow between different input states by giving mouse, keyboard, touchscreen, and voice reader users the choice to select/deselect benefits both efficiency and accessibility.
  • One positive accessibility impact of radio button selection method, particularly for Amazon Checkout and Outlook settings is a clear focus box for keyboard users to ensure they know which radio button they are selecting.
  • A negative accessibility impact of auto-selection using arrow keys is that users who accidentally press an arrow key may change their selection without realizing it.
  • Mouse and keyboard users seem to get the most prioritization, which touchscreen and users can face some inconsistencies. While screen-readers seem to be giving user all the necessary information, the organization of that information seem to be more of an afterthought. This could affect the user experience for those who may rely on an input method and make certain applications harder for them to use.
